# Curated /etc/* persist not restored on boot (secure profile)

## Parent

`.scratch/impermanence/PRD.md`

## Discovery

Found doing the `headless/secure` gate of
`.scratch/vm-profile-harness/issues/07-human-vm-smoke.md` (2026-06-12).
On a fresh `vm.sh --profile headless/secure` install, after reboot the
post-reboot checklist fails: **no SSH host keys**, `/persist` effectively
empty, curated Persist Mounts not active. SOPS happens to still work only
because its age key was baked into install-time state.

A precursor systemd `.mount` naming bug was already fixed locally
(commit `b4f2892`, NOT pushed). That fix is necessary but NOT sufficient;
the defects below remain.

## Symptoms (VM-observed, origin/main @530a9c6 + b4f2892)

- `ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub` → No such file.
- `ls /persist` → only `root/`; `/persist/etc` does not exist.
- `systemctl status etc-ssh.mount` → `Loaded: loaded` (good, post-fix) but
  `disabled` + `Active: inactive (dead)`; `What=/persist/etc/ssh` absent.
- `zfs mount` → `rpool/ROOT/arch` at `/`, `rpool/persist` at `/persist`
  AND at `/root`. **`rpool/ROOT/etc` (and opt/srv/usrlocal) are NOT
  mounted anywhere.** Only `/root` persists.

## Root causes (two, independent)

1. **Rollback datasets are never mounted.**
   `_impermanence_create_rollback_datasets` does `zfs create -o
   canmount=noauto` for etc/opt/srv/usrlocal and nothing ever mounts them.
   So `/etc` lives on the root dataset `rpool/ROOT/arch`; the initramfs
   `zfs-rollback` hook rolls back `rpool/ROOT/etc@blank` — an unmounted,
   empty dataset — a **no-op for the real `/etc`**. Impermanence-of-/etc
   does not function; only `/root` (active bind from `rpool/persist`) does.

2. **Curated `/etc/*` dirs are never staged into `/persist`.**
   `_impermanence_apply_curated` stages each CURATED_DIRS/FILES entry only
   `if [[ -e "${ROOT:-}$target" ]]`. `/etc/ssh`, `/etc/secrets`,
   `/etc/NetworkManager/system-connections`, `/etc/sudoers.d` likely don't
   exist yet when the Chroot Module runs (openssh + the SOPS program create
   them later), so they're silently skipped → `/persist/etc/*` never
   created → `etc-ssh.mount` (`What=/persist/etc/ssh`) has no source and
   stays inactive/disabled.

## Open design question (triage)

Is `/etc` meant to be its own rolled-back dataset (then the create step
must mount it before staging + `@blank`), or has the design moved to a
root-dataset `/etc` persisted purely via curated bind-mounts (then the
ROLLBACK_DATASETS `etc/opt/srv/usrlocal` entries + the rollback hook are
vestigial and should go)? Needs an ADR decision before implementing.
Either way the curated-staging ordering (cause 2) must be fixed so the
persisted dirs exist on `/persist` when `@blank` is taken.

## Acceptance criteria

- [ ] Design decision recorded (dataset-rollback vs bind-only) in the
      impermanence ADR.
- [ ] After `vm.sh --profile headless/secure` + reboot: `/etc/ssh` host
      keys present and **identical across a second reboot**.
- [ ] `/persist/etc/ssh` (+ `/etc/secrets`, NM connections, sudoers.d)
      populated; their Persist Mounts `active`.
- [ ] Curated `/etc/*` survives the `@blank` rollback (whatever mechanism).
- [ ] bats coverage that would catch this without a VM (e.g. assert curated
      dirs are staged to `/persist` even when created post-Chroot-Module;
      assert rollback datasets are actually mounted if the design keeps
      them).
- [ ] `headless/secure` post-reboot checklist in issue 07 passes.

### 2026-06-12 — triage → ready-for-agent

> *This was generated by AI during triage.*

The "open design question" above is **resolved by ADR 0008**: `/etc` (and
`opt/srv/usrlocal`) ARE dedicated Rollback Datasets that must mount "via
standard ZFS `mountpoint` properties + systemd's `zfs-mount-generator`"
(§ Consequences), and `/etc/ssh` + `/etc/secrets` MUST persist because the
SOPS Runtime Service derives from them (§ Consequences, final bullet).
So this is an **implementation-vs-ADR-0008 bug**, not a design decision.

## Agent Brief

**Category:** bug
**Summary:** Impermanence never actually persists curated `/etc/*` (SSH
host keys, SOPS age key, NM connections, sudoers.d) nor mounts the
Rollback Datasets — a fresh `headless/secure` install boots with no host
keys. Make the implementation satisfy ADR 0008.

**Current behavior:**
On a fresh impermanence install (e.g. `vm.sh --profile headless/secure`),
after reboot:
- The Rollback Datasets (`rpool/ROOT/{etc,opt,srv,usrlocal}`) are created
  but **never mounted** (created `canmount=noauto`, no `zfs-mount-generator`
  / `zfs-list.cache` wiring). `/etc` therefore lives on the non-rolled-back
  root dataset, and the initramfs Rollback Hook's `zfs rollback …@blank`
  acts on an empty, unmounted dataset — a no-op for the real `/etc`.
- The Curated Persist Defaults under `/etc` (`/etc/ssh`, `/etc/secrets`,
  `/etc/NetworkManager/system-connections`, `/etc/sudoers.d`) are **not
  staged onto the Persist Dataset**. The Chroot Module stages each curated
  path only when it already exists on disk, but these are created later
  (openssh + the SOPS program), so the existence guard skips them. Their
  per-path Persist Mounts then have a non-existent `What=` source and stay
  inactive. Only `/root` (which exists when the module runs) persists.
- Net: no SSH host keys after reboot; `/persist` holds only `root/`.

**Desired behavior (per ADR 0008):**
- Each Rollback Dataset is mounted at its mountpoint at boot (before early
  services need `/etc`), so the `@blank` rollback governs the real path.
- Every Curated Persist Default — including paths created after the Chroot
  Module runs — ends up staged on the Persist Dataset and bind-mounted back
  over the rolled-back location, so SSH host keys + the SOPS age key survive
  the rollback. Specifically, after a fresh secure install + reboot the host
  keys exist and are **identical across a second reboot**.

**Key interfaces / contracts:**
- `_impermanence_create_rollback_datasets` (Chroot Module) — datasets must
  end up actually mounted (e.g. proper `canmount`/mountpoint + the
  `zfs-list.cache` that `zfs-mount-generator` consumes), not `noauto` with
  no generator.
- `_impermanence_apply_curated` — staging must not silently skip curated
  paths that don't yet exist when it runs. Decide and implement the
  ordering fix: either run curated staging after the programs that create
  those paths, or ensure the paths exist (create + populate) before
  `@blank` is taken. The `@blank` snapshot must still be genuinely blank of
  the staged content.
- `imp_write_mount_unit` / Persist Mount naming — already corrected to the
  systemd `.mount` contract (`<esc>.mount`, not `persist-<esc>.mount`) in
  **local commit `b4f2892` (NOT pushed)**; build on it / land it.
- ADR 0008 § Consequences says operators discover mounts via
  `systemctl list-units 'persist-*.mount'` — that glob is now wrong (the
  units are `<esc>.mount`; the `status` verb filters by `What=` under the
  Persist Dataset). **Update ADR 0008 to match.**

**Acceptance criteria:**
- [ ] After `vm.sh --profile headless/secure` + reboot, `/etc/ssh` host
      keys exist and are identical across a second reboot.
- [ ] `/persist/etc/{ssh,secrets}` + NM connections + sudoers.d populated;
      their Persist Mounts are `active`.
- [ ] Rollback Datasets are mounted at boot (verify the `@blank` rollback
      affects the live `/etc`, not an unmounted dataset).
- [ ] SOPS Runtime Service still green after reboot (depends on the above).
- [ ] bats coverage that fails without a VM: assert curated paths are staged
      to the Persist Dataset even when created after the Chroot Module; and
      assert Rollback Datasets are configured to mount (not `noauto`-orphan).
- [ ] ADR 0008 updated re: Persist Mount naming + discovery.
- [ ] `headless/secure` post-reboot checklist in
      `.scratch/vm-profile-harness/issues/07-human-vm-smoke.md` passes
      (final boot proof runs through that human VM gate).

**Out of scope:**
- The systemd `.mount` naming fix itself (done in `b4f2892`).
- Redesigning the persist/rollback model — ADR 0008 stands; this is about
  conforming the implementation to it.
- Non-`/etc` rolled-back paths beyond confirming they mount.

**Verification note:** code + bats are agent-doable; the authoritative
boot/persistence proof requires the libvirt VM gate (issue 07), which an
AFK agent typically can't run — leave that final tick to the human gate.

> *This was generated by AI during triage.*

Implemented Approach A + the issues each step exposed. Suite 1010→1008
green throughout. **Local commits only (not pushed):**

- `b4f2892` Name persist `.mount` units after `Where=` (systemd contract).
- `0c81690` Create Rollback Datasets early (`canmount=on`) in the layout
  phase (`imp_create_rollback_datasets`, called from `_create_os_datasets`;
  removed the late `canmount=noauto` creation from the Chroot Module). Fixes
  cause #1 — datasets now mount.
- `b2d3bd7` `persist_stage_in_move`: move CONTENTS when the source is a
  mountpoint (`/root` is both a Rollback Dataset + curated dir → `mv`
  EBUSY).
- `ca5ae0d` Create the Persist Dataset early too (`imp_create_persist_
  dataset`) so it lands in the zfs-list.cache + mounts before
  local-fs.target.

**VM result (headless/secure, libvirt):** install completes; encrypted
root unlocks; **the dbus restart-loop is gone**; `/etc` now mounts from
`rpool/ROOT/etc` (verified `canmount=on`/`mounted=yes` pre-fix it was
`noauto`/no). System boots through `Mount ZFS filesystems` → `First Boot
Complete` → sshd socket. **Big progress.**

**Remaining layer (NOT fixed):** the boot lands in `systemd-firstboot`
("Initial Setup") and then hangs short of a usable login, because
`/etc/machine-id` is uninitialised after the `@blank` rollback. machine-id
(and other early-read curated FILES — `/etc/hostname`, `locale.conf`,
`vconsole.conf`, `fstab`, `adjtime`, the `CURATED_FILES` set) are read by
PID 1 / early boot BEFORE any `.mount` unit, so they CANNOT be restored via
the `/persist` bind mounts. They must instead live in the rolled-back
dataset's `@blank` with real install-time values. Proposed next step:
- Initialise `/etc/machine-id` (e.g. `systemd-machine-id-setup`) in the
  install BEFORE `@blank`, so the rollback restores a real, stable id.
- Stop staging `CURATED_FILES` to `/persist` (keep them in `@blank`,
  frozen-at-install). Keep the `/persist` bind treatment only for the
  `CURATED_DIRS` (ssh, secrets, NM connections, sudoers.d, pacman.d, root),
  which are read after `local-fs.target`.
- Then re-verify host-key persistence across a reboot on the secure VM.

**VM result (headless/secure):** install OK → encrypted unlock → **boots

6.18 LTS getty). `/persist/etc` is **fully populated** (ssh + real host
keys, secrets, machine-id, hostname, …). `/etc` mounts from
`rpool/ROOT/etc`. Major milestone — was "persisted nothing".

**THE ONE REMAINING LAYER — curated Persist Mounts don't auto-activate at
boot.** `etc-ssh.mount` is `Loaded: loaded` (naming OK) but `disabled` +
`inactive (dead)` after boot, so `/etc/ssh` stays empty → **host keys still
absent at login**. Proof it's purely auto-activation, not the mount:
`systemctl start etc-ssh.mount` → `active`, and `/etc/ssh` immediately shows
`ssh_host_ed25519_key` etc. The wants-symlinks all exist in
`/usr/lib/systemd/system/local-fs.target.wants/` (etc-ssh, etc-secrets,
etc-machine\x2did, root, …). So units + data + binds are correct; systemd
just never starts them at boot.

Hypotheses (not yet confirmed — no persistent journal on the VM,
`journalctl -b` = "No journal files"):
- Ordering interaction with the zfs-mount-generator units for `/etc` +
  `/persist` (the binds are `Before=local-fs.target` + auto
  `After=`/`RequiresMountsFor=` those zfs mounts → possible cycle, systemd
  drops them). Try: drop `Before=local-fs.target`, keep
  `RequiredBy=local-fs.target` (let auto After= order them); or
  `DefaultDependencies=no` + explicit `After=` the dataset mounts.
- **Likely-related clue:** `root.mount` NAME COLLISION — the curated bind
  for `/root` (`What=/persist/root`, `Where=/root`) is named `root.mount`,
  but zfs-mount-generator ALSO emits `root.mount` for the now-`canmount=on`
  `rpool/ROOT/root` (`Where=/root`). The generator unit (in
  `/run/systemd/generator`) shadows the `/usr/lib` bind → `/root` gets the
  empty rolled-back dataset, not the persist bind. Any `CURATED_DIR` that is
  also a Rollback-Dataset mountpoint (only `/root`) collides. Resolve the
  `/root` overlap (e.g. don't make `rpool/ROOT/root` a generator-mounted
  dataset, or rename, or drop `/root` from one of the two lists).

Next: confirm the auto-activation cause (needs a working journal or
`systemctl show etc-ssh.mount -p After,RequiredBy,WantedBy` analysis),
apply the ordering fix, re-verify host keys persist across TWO reboots.

### 2026-06-13 — CORE FIXED: host keys persist. Two more commits.

> *This was generated by AI during triage.*

The auto-activation cause was an **ordering cycle**:
`systemd-tmpfiles-setup.service` is `After=local-fs.target`, so the bind
units' `After=systemd-tmpfiles-setup.service` + `Before=local-fs.target`

- `d3f850e` Order the persist binds `After=zfs-mount.service` (not tmpfiles)

- `e59e1a5` Bootstrap now STAGES (copies) `/etc/systemd/system` +
  `/etc/tmpfiles.d` onto `/persist`, so the bind exposes install-time service

### 2026-06-13 — sops-runtime auto-start fixed (vendor wants-symlink)

> *This was generated by AI during triage.*

The last remaining slice (sops-runtime not auto-pulled into the boot
transaction) is resolved in code + bats.

**Root cause:** `sops/install.sh` enabled the service with
`systemctl enable sops-runtime.service`, which writes the wants-symlink to
`/etc/systemd/system/sysinit.target.wants/`. `/etc` is a Rollback Dataset
(rolled back to `@blank`) AND bind-covered by the Persist Dataset, so that
symlink is not reliably visible to PID 1 when it computes the boot
transaction for this EARLY unit (`WantedBy=sysinit.target`,
`Before=sysinit.target`, `DefaultDependencies=no`). The unit was
`is-enabled` and ran on demand (`systemctl start` → `active`/SUCCESS) but
never auto-fired. Every working enablement in the impermanence design uses
`/usr/lib/systemd/system/<target>.wants/` (curated mounts via
`imp_link_wants`) — the never-rolled-back, never-bind-covered root dataset,
always read at the earliest transaction. sops-runtime was the lone
`systemctl enable` (→ `/etc`) holdout.

**Fix:**
- New `programs/security/sops/scripts/enable-runtime.sh` →
  `sops_enable_runtime [root]`: writes
  `/usr/lib/systemd/system/sysinit.target.wants/sops-runtime.service` →
  `../sops-runtime.service` (idempotent; root-prefixable for tests). This is
  the canonical vendor-enable path and matches the curated-mount convention.
- `install.sh` sources the helper and calls it instead of `systemctl enable`.
- `tests/sops.bats` +3: asserts the `/usr/lib` symlink + target, asserts NO
  `/etc/.../sysinit.target.wants/` symlink (the rolled-back trap), and
  idempotency. Ordering was never the problem — sops-runtime is
  `After=local-fs.target` and the age key lives under `/etc/secrets`
  (`etc-secrets.mount`, `Before=local-fs.target`), so inputs are present once
  it is pulled in.

shellcheck clean (`-x -P SCRIPTDIR`); full bats green (1012).
