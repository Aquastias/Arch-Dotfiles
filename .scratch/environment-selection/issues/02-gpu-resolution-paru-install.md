# GPU Resolution + paru install step

Status: ready-for-agent
Category: enhancement

## Parent

## What to build

Implement the GPU Resolution function in the config loading module. It accepts the raw `"gpu"` value from the Install Config and produces two lists: `GPU_PACMAN_PACKAGES` (fed into `packages.groups.gpu` before pacstrap) and `GPU_PARU_PACKAGES` (AUR packages installed via paru in the profiles phase).

When `"gpu"` is `"auto"`, detect GPU vendors from `lspci` on the live ISO. The `lspci` call must be an injectable seam so tests can control its output. Normalise the result to an array. Map each vendor to its package set:

- `amd` → `vulkan-radeon xf86-video-amdgpu mesa libva-mesa-driver`
- `nvidia` → `nvidia-open-dkms nvidia-utils lib32-nvidia-utils libva-nvidia-driver egl-wayland`
- `intel` → `intel-media-driver` (Broadwell+ device ID) or `libva-intel-driver` (pre-Broadwell), selected by lspci device ID
- VM GPU (VMware / VirtualBox / virtio-gpu strings in lspci output) → `mesa` only; log a notice, do not abort

When the resolved vendor array contains both `"amd"` and `"nvidia"` (hybrid), add `envycontrol` to `GPU_PARU_PACKAGES`.

Add a step in the profiles phase: after paru is bootstrapped for the primary user and before user programs are installed, run `paru -S` for any entries in `GPU_PARU_PACKAGES`. This is a targeted addition to the profiles phase, not a general mechanism.

## Agent Brief

**Category:** enhancement
**Summary:** Implement GPU Resolution function and paru install step for AUR GPU packages.

**Current behavior:**
`packages.groups.gpu` exists in `install.jsonc` but is always an empty list — no logic populates it. GPU drivers must be installed manually after first boot.

**Desired behavior:**
A GPU Resolution function runs at config-load time (after validation, before layout planning). It reads the normalised `"gpu"` value from the Install Config and populates two variables: `GPU_PACMAN_PACKAGES` (fed into `packages.groups.gpu` before pacstrap) and `GPU_PARU_PACKAGES` (installed via paru in the profiles phase).

If `"gpu"` is `"auto"`, call an injectable `lspci` seam function to get the raw lspci output, then detect GPU vendors from known strings. All other values are taken directly from config. Multiple vendors (explicit array or auto-detected hybrid) produce the union of their package sets.

Vendor → `GPU_PACMAN_PACKAGES` mapping:
- `amd` → `vulkan-radeon xf86-video-amdgpu mesa libva-mesa-driver`
- `nvidia` → `nvidia-open-dkms nvidia-utils lib32-nvidia-utils libva-nvidia-driver egl-wayland`
- `intel Broadwell+` → `intel-media-driver` (detect by PCI device ID ≥ 0x1600)
- `intel pre-Broadwell` → `libva-intel-driver`
- VM GPU (VMware/VirtualBox/virtio-gpu detected in lspci output) → `mesa`; log a notice, do not abort

Hybrid (both amd and nvidia resolved) → add `envycontrol` to `GPU_PARU_PACKAGES`.

Profiles phase addition: after paru is bootstrapped for the primary user (first user in host config), install any entries in `GPU_PARU_PACKAGES` via `paru -S --noconfirm --needed` before user programs run. This is a targeted step, not a general mechanism.

**Key interfaces:**
- `resolve_gpu_packages()` — reads normalised gpu array, populates `GPU_PACMAN_PACKAGES` and `GPU_PARU_PACKAGES`. Call site: config loading, after validation.
- `_gpu_lspci_output()` — injectable seam that wraps `lspci`. In production: `lspci`. In tests: overridden to return controlled strings.
- `packages.groups.gpu` — populated from `GPU_PACMAN_PACKAGES` before pacstrap; existing pacstrap logic already reads this field.
- Profiles phase — new step: if `GPU_PARU_PACKAGES` non-empty, run `arch-chroot /mnt su - <primary_user> paru -S --noconfirm --needed ${GPU_PARU_PACKAGES[@]}` after paru bootstrap and before user programs.
- BATS test file — new. Uses `_gpu_lspci_output` override to simulate hardware. Prior art: `iso-resolver.bats` overrides `_iso_resolver_resolve_url` as a seam.
