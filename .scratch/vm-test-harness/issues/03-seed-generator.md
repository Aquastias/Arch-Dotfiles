# Cloud-init seed generator module

Status: ready-for-agent
Category: enhancement

## Parent

`.scratch/vm-test-harness/PRD.md`

## What to build

A deep, testable module that produces a cloud-init NoCloud seed ISO containing a `user-data` document parameterised by repo URL and test hostname.

Public interface: one function that takes a repo URL, a test hostname, and an output directory; produces `seed.iso` in that directory; prints the absolute path of the produced file on stdout.

The `user-data` content the seed must encode:

- Cloud-config format (`#cloud-config` header).
- A single `runcmd` block that, in this order:
  1. Redirects stdout/stderr to `/dev/ttyS0`.
  2. Removes any prior clone in `/root`.
  3. `git clone`s the supplied repo URL into `/root`.
  4. Changes into the cloned repo's `.os/` directory.
  5. Edits `install.jsonc` so that the system hostname becomes the supplied test hostname (replacing the empty default).
  6. Runs `install.sh --unattended`.
  7. Captures the exit code.
  8. Emits a sentinel line of the exact form `===INSTALLER-EXIT-N===` (where `N` is the captured exit code) to the serial console.
  9. `sync` and `poweroff`.

The seed ISO is built using `cloud-localds` (from `cloud-image-utils`). The module is responsible for invoking it and asserting the output exists, but it is *not* responsible for installing the dependency — that lives in the orchestrator's dependency check.

The repo URL and hostname must be substituted into the template such that the generated `user-data` contains literal values, not placeholders.

The module must be sourceable in isolation and must not depend on libvirt.

## Acceptance criteria

- [ ] A new module under `.os/lib/` exposes a single public function with the contract above
- [ ] The generated `seed.iso` exists at the returned path
- [ ] The generated `user-data` (verified by mounting the ISO or by exposing it as an internal artefact for tests) contains the literal repo URL and test hostname — no remaining placeholders
- [ ] The generated `runcmd` redirects to `/dev/ttyS0` and ends with the sentinel + `poweroff`
- [ ] The sentinel line format is exactly `===INSTALLER-EXIT-N===` (no surrounding text on that line)
- [ ] If `cloud-localds` is missing, the function exits non-zero with a clear message (it does not attempt to install it)
- [ ] Bats tests under `.os/tests/` cover: substitution correctness (URL + hostname appear), runcmd shape (redirect, sentinel, poweroff present), and missing-`cloud-localds` failure mode
- [ ] All bats tests pass
- [ ] Shellcheck passes
